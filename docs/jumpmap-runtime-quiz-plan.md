# 점프맵-퀴즈코어 통합 개발계획 (운영안)

작성일: 2026-02-21  
상태: 운영안(확정 전)  
범위: 점프맵 런타임 + 퀴즈코어 연결, 멀티플레이(2~6인), 전자칠판 운영

---

## 1) 목표
- 전자칠판 분할 구역(2~6인)에서 각 플레이어가 독립적으로 점프맵을 진행한다.
- 퀴즈 버튼으로 퀴즈를 풀고, 결과에 따라 게이지를 회복한다.
- 현재 맵 데이터/에셋/로컬 개발 흐름을 유지한다.

---

## 2) 핵심 원칙
- 기본 게임 모드는 공통(`globalMode`)으로 사용한다.  
  기본값: `점프맵`.
- 플레이어 상태는 `playerId` 단위로 완전 분리한다.
- 게이지 시스템은 `점프맵 런타임` 소유, 퀴즈코어는 문제/채점만 담당한다.
- 런타임-퀴즈 연결은 브리지 계층에서만 수행한다.
- 저장 맵 포맷(`version:2`) 호환성을 깨지 않는다.
- 로컬 우선 원칙을 유지한다(서버/DB 비필수).

---

## 3) 비범위(현 단계)
- 신규 게임 모드 구현(`jumpmap` 외) 자체는 본 단계 범위에서 제외한다.
- 에디터 UI 대규모 리디자인은 제외한다.
- 물리 시스템 전면 교체(엔진 교체 포함)는 제외한다.

---

## 4) 모듈 경계
- `jump-runtime`
  - 이동/점프/충돌/카메라/게이지
  - 플레이어별 런타임 상태 소유
- `quiz-core`
  - 문제 출제/정답 판정/점수 계산
- `integration-bridge`
  - 퀴즈 요청/응답 중계
  - 상태 전이 트리거(`PLAYING <-> QUIZ_OPEN <-> QUIZ_RESULT`)
  - 런타임 상태 오염 방지용 입력 게이팅
- `mode-registry`
  - 현재: `basic`, `jumpmap`
  - 추후: 추가 게임 모드 등록
- `game-rule-adapter` (모드별 정책 어댑터)
  - 점프맵: 게이지 소모/회복/차단 규칙
  - 추후 게임: 다른 리소스(예: 체력, 스킬포인트, 탄약 등) 규칙
  - 공통 인터페이스로 런타임에 정책만 주입

---

## 5) 브리지 이벤트 계약(고정 필드)
- 필수 공통 필드:
  - `playerId`, `zoneId`, `timestamp`, `source`
- 런타임 -> 브리지:
  - `quiz:open-request` `{ playerId, zoneId, reason, timestamp }`
  - `gauge:consume` `{ playerId, zoneId, amount, reason, timestamp }`
- 브리지 -> 런타임:
  - `quiz:resolved` `{ playerId, zoneId, correct, refillAmount, questionId, timestamp }`
  - `quiz:close` `{ playerId, zoneId, next: 'PLAYING' | 'QUIZ_OPEN', timestamp }`
- 확장 필드(모드 공용):
  - `resourceKey` (예: `gauge`, `stamina`, `energy`)
  - `resourceDelta`
  - `ruleSetId` (어떤 게임 규칙셋인지)

규칙:
- 계약 필드 누락 이벤트는 `drop` 처리하고 `warn` 로그를 남긴다.
- 계약 필드 누락 이벤트는 카운터(`bridgeDroppedEvents`)를 누적한다.
- 런타임이 퀴즈코어 내부 함수/상태를 직접 참조하지 않는다.
- 퀴즈코어는 `resourceKey/resourceDelta` 계산을 하지 않는다(모드별 어댑터 책임).

---

## 6) 상태 머신
- 플레이어별 상태:
  - `PLAYING`
  - `QUIZ_OPEN`
  - `QUIZ_RESULT`
  - `PAUSED`
  - `FINISHED`
- 전이 규칙:
  - `PLAYING -> QUIZ_OPEN`: 퀴즈 버튼
  - `QUIZ_OPEN -> QUIZ_RESULT`: 채점 완료
  - `QUIZ_RESULT -> PLAYING`: "맵으로 복귀"
  - `QUIZ_RESULT -> QUIZ_OPEN`: "다음 문제"
  - `PAUSED`는 해당 플레이어 입력만 정지(다른 플레이어 비정지)

---

## 7) 게이지 규칙(초기값)
- 이동: 소모
- 점프: 소모
- 공중 이동: 게이지 0이어도 허용
- 더블점프: 게이지 부족 시 불가
- 정답: 큰 회복
- 오답: 회복 없음(`+0`)
- 오답 결과 표시 후 `3초 딜레이` 적용

주의:
- 지상 차단 규칙과 공중 허용 규칙을 분리해 고착 버그를 방지한다.
- 게이지 0일 때 입력 무시/차단 여부는 지상 상태에서만 적용한다.

확장 원칙:
- 위 규칙은 `jumpmap` 전용 기본값이다.
- 다른 게임 모드는 동일 구조를 재사용하되, 리소스 키/소모·보상 규칙만 어댑터에서 교체한다.

## 8) 멀티플레이 운영 규칙
- 모든 플레이어는 기본적으로 같은 게임 모드 사용
- 구역 밖 입력 무시
- 퀴즈는 `개인 세션`으로만 진행
- 한 플레이어가 퀴즈를 열어도 다른 플레이어는 계속 진행 가능(기본값)
- 플레이어명/색상 라벨은 플레이 중 항상 표시
- 플레이어별 퀴즈 세션/게이지/입력 상태는 절대 공유 금지
- 각 문제 종료 시(정답/오답 표시 후) 해당 플레이어에게만 선택 팝업 표시:
  - `다음 문제`
  - `맵으로 복귀`

## 10) 저장 호환성 규칙
- 맵 스키마는 `version:2` 유지.
- 통합 기능 상태는 맵 본문이 아닌 런타임 세션 상태로 분리 저장.
- 기준 회귀 파일:
  - `save_map/jumpmap-20260219-164307.json`
  - `save_map/jumpmap-20260221-132734.json`

추가 원칙:
- 퀴즈코어 공통 데이터(문항, 판정 결과)와 게임모드 전용 상태(게이지 등)는 저장 경계를 분리한다.
- 모드 추가 시에도 기존 맵 JSON을 수정하지 않고, 런타임 세션 스키마 확장으로 대응한다.

## 11) 단계별 구현
1. `Phase A`: 브리지 이벤트 계약 고정
2. `Phase B`: 플레이어별 게이지 상태 분리
3. `Phase C`: 퀴즈 진입/결과/복귀 루프 연결
4. `Phase D`: 멀티플레이(2~6인) 독립성 검증
5. `Phase E`: 저장/불러오기 회귀 검증
6. `Phase F`: UI/운영 설정(공통 모드, 옵션) 마감
7. `Phase G`: 모드 어댑터 추출(점프맵 외 1개 목업 모드로 검증)

각 단계 통과 기준:
- 코드 체크(`node --check`) 통과
- 체크리스트 섹션 통과
- 회귀 이슈 재발 없음
